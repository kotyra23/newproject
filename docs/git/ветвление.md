# 🌿 Git: Ветвление и Слияние

## 📖 Зачем нужны ветки?

**Ветка (branch)** — это независимая линия разработки.  
Позволяет работать над новыми функциями, исправлять баги или экспериментировать, не затрагивая основную версию кода.

### Типичный рабочий процесс:

main (стабильная версия)
│
├─► feature/login ← новая функция
├─► bugfix/header ← исправление бага
└─► experiment/new-ui ← эксперимент


---

## 🔄 Работа с ветками

### Создание и переключение

```bash
# Создать новую ветку:
git branch feature-login

# Переключиться на ветку:
git checkout feature-login

# Создать и сразу переключиться (короткая запись):
git checkout -b feature-login

# Современный синтаксис (Git 2.23+):
git switch -c feature-login

Просмотр веток

# Список локальных веток:
git branch

# Список всех веток (локальные + удалённые):
git branch -a

# Детальная информация:
git branch -v

# Показать граф веток:
git log --oneline --graph --all

Удаление веток

# Удалить локальную ветку:
git branch -d feature-login      # Безопасно (только если слита)
git branch -D feature-login      # Принудительно

# Удалить удалённую ветку:
git push origin --delete feature-login

🔀 Слияние веток (Merge)
Базовый merge

# 1. Переключиться на целевую ветку (куда сливаем):
git checkout main

# 2. Слить изменения из другой ветки:
git merge feature-login

Типы слияния

| Тип | Описание | Когда используется |
| :--- | :--- | :--- |
| `Fast-forward` | Просто перемещает указатель, если нет расхождений | Ветка создана от актуальной `main` и не было параллельных коммитов |
| `Three-way merge` | Создаёт новый "коммит слияния" | Были параллельные изменения в обеих ветках |
| `No-ff merge` | Принудительно создаёт коммит слияния | Для сохранения истории ветки (`git merge --no-ff`) |

Пример с конфликтом

# Если Git не может автоматически слить изменения:
CONFLICT (content): Merge conflict in file.txt

# 1. Открыть файл и найти маркеры конфликта:
<<<<<<< HEAD
Код из текущей ветки (main)
=======
Код из сливаемой ветки (feature)
>>>>>>> feature-login

# 2. Вручную отредактировать файл, оставив нужный код

# 3. Отметить разрешение конфликта:
git add file.txt

# 4. Завершить слияние:
git commit -m "fix: разрешён конфликт в file.txt"

# Или отменить слияние:
git merge --abort

🔁 Rebase (Перебазирование)
Rebase — переносит коммиты одной ветки на конец другой, создавая линейную историю.

Базовый синтаксис

# Перебазировать текущую ветку на main:
git checkout feature-login
git rebase main

Визуальное сравнение

До rebase:          После rebase:
main:  A---B        main:  A---B---C'---D'
                \                    \
feature: C---D     feature:          (слита)


Интерактивный rebase (для очистки истории)

# Редактировать последние 3 коммита:
git rebase -i HEAD~3

Команды в интерактивном режиме:

| Команда | Описание |
| :--- | :--- |
| `pick` | Использовать коммит как есть |
| `reword` | Использовать коммит, но изменить сообщение |
| `edit` | Остановиться для правки коммита |
| `squash` | Объединить с предыдущим коммитом |
| `drop` | Удалить коммит |

⚠️ Золотое правило rebase
❌ Никогда не делай rebase публичных веток, которые используют другие разработчики.
✅ Rebase только локальные ветки перед слиянием в main.

🆘 Разрешение конфликтов: Пошагово
Шаг 1: Обнаружение

git merge feature-login
# Output: CONFLICT (content): Merge conflict in config.py

Шаг 2: Проверка статуса

git status
# Покажет: "both modified: config.py"

Шаг 3: Открытие файла и правка

# Найди маркеры в файле:
<<<<<<< HEAD
database_url = "postgresql://prod-db"
=======
database_url = "postgresql://dev-db"
>>>>>>> feature-login

# Оставь нужный вариант или объедини:
database_url = "postgresql://prod-db"  # Оставили продакшен

Шаг 4: Завершение

# Отметить файл как разрешённый:
git add config.py

# Завершить слияние:
git commit -m "fix: разрешён конфликт в config.py"

Отмена слияния (если всё плохо):

git merge --abort

#📦 Stash: Временное сохранение
Stash — откладывает изменения, чтобы переключиться на другую задачу.

# Сохранить изменения (рабочая директория станет чистой):
git stash

# Сохранить с сообщением:
git stash save "WIP: работа над логикой оплаты"

# Посмотреть список stash'ей:
git stash list

# Применить последний (не удаляя из списка):
git stash apply

# Применить и удалить:
git stash pop

# Применить конкретный:
git stash apply stash@{2}

# Удалить stash:
git stash drop stash@{1}

# Очистить все:
git stash clear

#📋 Шпаргалка: Ветвление

| Задача | Команда |
| :--- | :--- |
| Создать ветку | `git branch name` |
| Переключиться | `git checkout name` / `git switch name` |
| Создать + переключиться | `git checkout -b name` |
| Список веток | `git branch` / `git branch -a` |
| Удалить ветку | `git branch -d name` |
| Слить ветку | `git merge name` |
| Rebase | `git rebase main` |
| Интерактивный rebase | `git rebase -i HEAD~3` |
| Сохранить изменения | `git stash` |
| Вернуть изменения | `git stash pop` |
| Отменить слияние | `git merge --abort` |
| Отменить rebase | `git rebase --abort` |

🧩 Продвинутые техники
Cherry-pick: Выбрать отдельный коммит

# Скопировать конкретный коммит в текущую ветку:
git cherry-pick <commit-hash>

# Несколько коммитов:
git cherry-pick <hash1> <hash2>

Отмена коммитов

# Отменить последний коммит, сохранив изменения:
git reset --soft HEAD~1

# Отменить коммит и изменения в файлах:
git reset --hard HEAD~1

# Создать новый коммит, отменяющий изменения старого:
git revert <commit-hash>

Поиск проблемного коммита (bisect)

# Запустить бинарный поиск:
git bisect start
git bisect bad HEAD          # Текущая версия — плохая
git bisect good v1.0         # Старая версия — хорошая

# Git будет переключать коммиты — тестируй и отмечай:
git bisect good   # Если всё ок
git bisect bad    # Если баг воспроизводится

# Завершить поиск:
git bisect reset

⚠️ Важные нюансы

Всегда делай git pull перед началом работы — чтобы избежать конфликтов.

Используй --no-ff для слияния фич — сохраняет историю: git merge --no-ff feature.

Rebase только локальные ветки — не трогай публичную историю.

Конфликты — это нормально — решай их спокойно, шаг за шагом.

Stash — не замена коммиту — используй для временных переключений.
