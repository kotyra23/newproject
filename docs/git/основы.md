# 🐙 Git: Основы

##  Что такое Git?

**Git** — это распределённая система контроля версий (DVCS).  
Она позволяет отслеживать изменения в файлах (обычно в коде), координировать работу между разработчиками и возвращаться к предыдущим версиям.

### 3 Состояния файла в Git

Важно понимать, что файл в Git может находиться в одном из трёх состояний:

1. **Modified (Изменён)** — файл изменён в рабочей директории, но не сохранён в базе Git.
2. **Staged (Проиндексирован)** — файл добавлен в индекс (staging area), готов к коммиту.
3. **Committed (Закоммичен)** — изменения надёжно сохранены в локальной базе данных Git.

## 🛠 Первоначальная настройка

Перед началом работы нужно представиться Git'у.

```bash
# Установить имя и email (будут видны в истории коммитов):
git config --global user.name "Nikita"
git config --global user.email "your@email.com"

# Установить редактор по умолчанию (например, Nano или VS Code):
git config --global core.editor "nano"

# Включить цветной вывод (удобно):
git config --global color.ui auto

# Проверить настройки:
git config --list

#🚀 Начало работы
1. Инициализация проекта

# Создать пустой репозиторий в текущей папке:
git init

# Клонировать существующий репозиторий с GitHub/GitLab:
git clone https://github.com/user/repo.git

2. Проверка статуса

Команда git status — твой лучший друг. Она показывает, что происходит.

git status

Пример вывода:

On branch main
Changes not staged for commit:
  (use "git add <file>..." to update what will be committed)
  (use "git restore <file>..." to discard changes in working directory)
        modified:   file.txt

Untracked files:
  (use "git add <file>..." to include in what will be committed)
        new_script.sh

#🔄 Базовый рабочий цикл
Шаг 1: Добавление в индекс (Staging)

Перед коммитом нужно выбрать, какие изменения попадут в историю.

# Добавить конкретный файл:
git add file.txt

# Добавить все изменённые и новые файлы (включая подпапки):
git add .

# Добавить всё (включая удалённые файлы):
git add -A

Шаг 2: Коммит (Сохранение)
Фиксируем изменения с комментарием.

# Коммит с сообщением в одну строку:
git commit -m "feat: добавлена новая функция логина"

# Коммит всех отслеживаемых изменений (пропускает git add):
git commit -am "fix: исправлена опечатка в конфиге"

Шаг 3: Отправка на сервер (Push)


# Добавить удалённый репозиторий (если клонировали, это уже сделано):
git remote add origin https://github.com/user/repo.git

# Отправить изменения в ветку main:
git push origin main

# Первый раз (связать локальную ветку с удалённой):
git push -u origin main

Шаг 4: Получение изменений (Pull)

Если над проектом работают другие (или ты с другого ПК):

# Скачать и сразу слить изменения:
git pull origin main

# Просто скачать без слияния (чтобы先看):
git fetch origin

#🔍 Просмотр истории

# Полная история:
git log

# Кратко (по одной строке на коммит):
git log --oneline

# С красивым графом веток:
git log --oneline --graph --all

# Посмотреть, что изменилось в конкретном коммите:
git show <hash-коммита>

#🚫 .gitignore
Файл .gitignore говорит Git'у, какие файлы не нужно отслеживать (логи, временные файлы, пароли).

# Игнорировать все файлы с расширением .log
*.log

# Игнорировать папку node_modules
node_modules/

# Игнорировать конкретный файл
secret.txt

#📋 Шпаргалка по основным командам

| Задача | Команда |
| :--- | :--- |
| Инициализация | `git init` |
| Клонирование | `git clone <url>` |
| Статус | `git status` |
| Добавить файл | `git add <file>` |
| Добавить всё | `git add .` |
| Коммит | `git commit -m "msg"` |
| История | `git log --oneline` |
| Отправить | `git push origin main` |
| Получить | `git pull origin main` |
| Игнорировать файл | `git rm --cached <file>` |

⚠️ Частые ошибки новичков

Забыл git add перед git commit — коммит создастся пустым.

Push rejected — значит, на сервере есть изменения, которых нет у тебя. Сначала сделай git pull.

Коммитит лишнее (например, node_modules или .env с паролями) — используй .gitignore.

Паника при конфликтах — это нормально, их нужно просто решить вручную.
