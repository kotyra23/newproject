# 🎓 Git: Продвинутые техники

## 🔙 Отмена изменений

### Разница между reset, revert, restore

| Команда | Что делает | Когда использовать |
|---------|-----------|-------------------|
| `git reset` | Перемещает указатель ветки, меняет индекс | Локально, до `push` |
| `git revert` | Создаёт новый коммит, отменяющий изменения | После `push`, в публичных ветках |
| `git restore` | Отменяет изменения в файлах (рабочая директория) | Когда нужно откатить правки в файле |

---

### 🔄 Git Reset

```bash
# Soft: отменяет коммит, сохраняет изменения в индексе
git reset --soft HEAD~1

# Mixed (по умолчанию): отменяет коммит и индекс, сохраняет файлы
git reset --mixed HEAD~1
git reset HEAD~1  # То же самое

# Hard: отменяет всё, удаляет изменения безвозвратно! ⚠️
git reset --hard HEAD~1

# Отменить reset (если не делал другие коммиты):
git reflog
git reset HEAD@{1}

↩️ Git Revert

# Отменить последний коммит (создаёт новый коммит-отмену):
git revert HEAD

# Отменить конкретный коммит по хешу:
git revert abc1234

# Отменить диапазон коммитов:
git revert abc1234..def5678

# Без автоматического коммита (чтобы поправить перед коммитом):
git revert -n HEAD

Преимущество revert:
✅ Безопасно для публичных веток
✅ Сохраняет историю изменений
✅ Можно откатить откат (revert of revert)

♻️ Git Restore / Checkout

# Отменить изменения в файле (вернуть к последнему коммиту):
git restore file.txt

# Отменить все изменения в рабочей директории:
git restore .

# Вернуть файл из конкретного коммита:
git restore -s abc1234 file.txt

# Старый синтаксис (всё ещё работает):
git checkout -- file.txt

🪝 Git Hooks
Hooks — скрипты, которые выполняются автоматически при определённых событиях.

.repo/
└── .git/
    └── hooks/
        ├── pre-commit      # Перед коммитом
        ├── commit-msg      # Проверка сообщения коммита
        ├── pre-push        # Перед отправкой на сервер
        └── post-merge      # После слияния

# Создать хук:
nano .git/hooks/pre-commit

#!/bin/bash
# .git/hooks/pre-commit

# Проверить синтаксис Python
echo "🔍 Проверка кода..."

# Запустить линтер
flake8 . --count --select=E9,F63,F7,F82 --show-source --statistics
if [ $? -ne 0 ]; then
    echo "❌ Ошибки линтера! Коммит отменён."
    exit 1
fi

# Проверить, что нет отладочных принтов
if grep -r "print('DEBUG" . --include="*.py" > /dev/null; then
    echo "❌ Найден отладочный print! Удали перед коммитом."
    exit 1
fi

echo "✅ Все проверки пройдены"
exit 0

# Сделать хук исполняемым:
chmod +x .git/hooks/pre-commit

Популярные хуки:

| Хук | Событие | Пример использования |
| :--- | :--- | :--- |
| `pre-commit` | Перед `git commit` | Линтеры, форматирование, тесты |
| `commit-msg` | После ввода сообщения | Проверка формата сообщения |
| `pre-push` | Перед `git push` | Запуск всех тестов перед отправкой |
| `post-merge` | После `git pull`/`merge` | Установка зависимостей, миграции БД |

📦 Submodules и Subtrees
Git Submodules
Submodule — вложение одного репозитория в другой.

# Добавить submodule:
git submodule add https://github.com/user/lib.git libs/mylib

# Клонировать репо с submodules:
git clone --recurse-submodules https://github.com/user/project.git

# Или после клонирования:
git submodule init
git submodule update

# Обновить submodule до последней версии:
cd libs/mylib
git pull origin main
cd ../..
git add libs/mylib
git commit -m "update submodule mylib"

# Добавить код другого репо как поддиректорию:
git subtree add --prefix=libs/mylib https://github.com/user/lib.git main --squash

# Обновить:
git subtree pull --prefix=libs/mylib https://github.com/user/lib.git main --squash

# Отправить изменения обратно:
git subtree push --prefix=libs/mylib https://github.com/user/lib.git main

| Критерий | Submodule | Subtree |
| :--- | :--- | :--- |
| Сложность | Выше | Ниже |
| Изоляция | Полная | Частичная |
| Обновление | `submodule update` | `subtree pull` |
| Отправка изменений | Сложнее | `subtree push` |

🔍 Git Bisect: Поиск багов
Bisect — бинарный поиск проблемного коммита.

# Запустить поиск:
git bisect start

# Указать "плохой" коммит (обычно HEAD):
git bisect bad HEAD

# Указать "хороший" коммит (где баг ещё не проявлялся):
git bisect good v1.0.0

# Git переключит на коммит посередине — тестируй:
# Если баг есть:
git bisect bad
# Если бага нет:
git bisect good

# Повторять, пока Git не найдёт проблемный коммит

# Завершить поиск:
git bisect reset

Автоматический bisect со скриптом:

git bisect start HEAD v1.0.0
git bisect run ./test-script.sh
# test-script.sh должен возвращать:
# 0 = хорошо, 1 = плохо, 125 = пропустить коммит

🏷️ Теги (Tags)
Создание тегов:

# Лёгкий тег (просто метка):
git tag v1.0.0

# Аннотированный тег (с сообщением, автором, датой):
git tag -a v1.0.0 -m "Релиз версии 1.0.0"

# Подписанный тег (GPG):
git tag -s v1.0.0 -m "Подписанный релиз"

Работа с тегами:

# Список тегов:
git tag
git tag -l "v1.*"  # Фильтрация по шаблону

# Информация о теге:
git show v1.0.0

# Отправка тегов:
git push origin v1.0.0          # Конкретный тег
git push origin --tags          # Все теги

# Удаление тега:
git tag -d v1.0.0                           # Локально
git push origin --delete v1.0.0             # Удалённо

🔧 Работа с историей
Изменение последнего коммита:

# Поменять сообщение:
git commit --amend -m "Новое сообщение"

# Добавить забытый файл в последний коммит:
git add forgotten.txt
git commit --amend --no-edit

# Поменять автора:
git commit --amend --author="Nikita <email@example.com>"

Переписать несколько коммитов (interactive rebase):

# Редактировать последние 5 коммитов:
git rebase -i HEAD~5

Пример сеанса:

pick abc1234 feat: добавить логин
pick def5678 fix: исправить баг
pick ghi9012 docs: обновить README

# Можно изменить на:
pick abc1234 feat: добавить логин
squash def5678 fix: исправить баг    # Объединить с предыдущим
reword ghi9012 docs: обновить README # Поменять сообщение

Очистка истории от чувствительных данных ⚠️

# Удалить файл из ВСЕЙ истории (осторожно!):
git filter-branch --force --index-filter \
  "git rm --cached --ignore-unmatch secret.txt" \
  --prune-empty --tag-name-filter cat -- --all

# Современный способ (быстрее):
git filter-repo --path secret.txt --invert-paths

# После очистки — принудительный пуш:
git push origin --force --all
git push origin --force --tags

#📋 Шпаргалка: Продвинутые команды

| Задача | Команда |
| :--- | :--- |
| Отменить коммит (локально) | `git reset --soft/hard HEAD~1` |
| Отменить коммит (публично) | `git revert <hash>` |
| Отменить изменения в файле | `git restore file.txt` |
| Создать хук | `nano .git/hooks/pre-commit` |
| Добавить submodule | `git submodule add <url> path` |
| Обновить submodules | `git submodule update --init --recursive` |
| Запустить bisect | `git bisect start` |
| Создать тег | `git tag -a v1.0 -m "msg"` |
| Изменить последний коммит | `git commit --amend` |
| Переписать историю | `git rebase -i HEAD~N` |
| Удалить файл из истории | `git filter-repo --path file --invert-paths` |

⚠️ Важные предупреждения

reset --hard безвозвратен — всегда делай бэкап ветки: git branch backup-branch.

Не меняй публичную историю — rebase / filter-branch только для локальных веток.

Hooks не копируются при клонировании — документируй их настройку в README.

Submodules требуют внимания — забывай обновлять = устаревший код.

Bisect меняет HEAD — не работай в других вкладках терминала во время поиска.
