# 🚨 Высокая нагрузка: Диагностика

## 📊 Быстрая диагностика (первый ответ)

```bash
# Общая картина:
top                    # Интерактивный мониторинг
htop                   # Улучшенный top (если установлен)
uptime                 # Load average за 1/5/15 мин

# Краткий обзор:
vmstat 1 5             # Статистика каждые 1 сек, 5 раз
iostat -x 1 5          # Статистика дисков

Что смотреть в top:

| Поле | Что означает | Тревожное значение |
| :--- | :--- | :--- |
| **load average** | Средняя нагрузка | `> числа ядер CPU` |
| **%CPU** | Загрузка процессора | `> 90%` постоянно |
| **%MEM** | Использование RAM | `> 80-90%` |
| **WA** (iowait) | Ожидание диска | `> 20-30%` |
| **SI/SO** | Swap in/out | Любое значение `> 0` |


#🔍 Детальная диагностика
1. Процессор (CPU)

# Топ процессов по CPU:
ps aux --sort=-%cpu | head -20

# Статистика по ядрам:
mpstat -P ALL 1

# Контекстные переключения:
vmstat 1 | grep -E "cs|in"

# Прерывания:
cat /proc/interrupts | tail -20

Признаки проблем:

us (user) высокий → приложение активно считает

sy (system) высокий → много системных вызовов

wa (iowait) высокий → узкое место в диске

id (idle) низкий → процессор загружен


2. Память (RAM)

# Детальная информация:
free -h
cat /proc/meminfo | grep -E "MemTotal|MemFree|MemAvailable|Buffers|Cached"

# Топ процессов по памяти:
ps aux --sort=-%mem | head -20

# Swap:
swapon --show
cat /proc/swaps


Признаки проблем:

MemAvailable < 10% → нехватка памяти

SwapUsed растёт → вытеснение в swap

si/so в vmstat > 0 → активный свопинг (тормоза!)

Быстрое освобождение памяти (кэш):

# Очистить page cache, dentries, inodes:
sudo sync && sudo sh -c 'echo 3 > /proc/sys/vm/drop_caches'

3. Дисковый ввод-вывод (I/O)

# Статистика по устройствам:
iostat -x 1 5

# Топ процессов по дисковой активности:
iotop -oPa              # Требует root и пакет iotop

# Очереди и задержки:
iostat -x | grep -E "Device|sda"

# Inodes и место:
df -i                   # Использование inodes
df -h                   # Использование места

4. Сеть

# Активные соединения:
ss -tulnp
netstat -tulpn          # Устаревший, но привычный

# Статистика по интерфейсам:
ip -s link show eth0
ifconfig eth0           # Устаревший

# Ошибки и дропы:
netstat -i
cat /proc/net/dev

# TCP-статистика:
netstat -st
ss -s


Признаки проблем:

Recv-Q / Send-Q не нулевые → буферы переполнены

retrans в netstat -st растёт → потери пакетов

drop в /proc/net/dev → проблемы с драйвером/железом


## 🛠 Практические сценарии
Сценарий 1: Сервер "тупит", load высокий


# 1. Смотрим load и uptime:
uptime

# 2. Если load > ядер CPU — ищем виновника:
ps aux --sort=-%cpu | head -10

# 3. Проверяем не диск ли узкое место:
iostat -x 1 3

# 4. Если iowait высокий — смотрим кто пишет:
iotop -oPa

# 5. Проверяем память:
free -h
vmstat 1 3

Сценарий 2: Приложение медленно отвечает

# 1. Проверить процесс приложения:
pgrep -a myapp
ps aux | grep myapp

# 2. Посмотреть открытые файлы/сокеты:
lsof -p <PID>
ss -tnp | grep <PID>

# 3. Проверить логи:
journalctl -u myapp -n 100 --no-pager
tail -f /var/log/myapp/error.log

# 4. Проверить системные вызовы (если нужно):
strace -p <PID> -c     # Статистика вызовов

Сценарий 3: Место на диске заканчивается


# 1. Найти большие папки:
du -h / | grep -E '^[0-9\.]+G' | sort -rh | head -20

# 2. Найти большие файлы:
find / -type f -size +500M -exec ls -lh {} \; 2>/dev/null

# 3. Проверить логи:
ls -lh /var/log/
journalctl --disk-usage

# 4. Очистить старые логи:
sudo journalctl --vacuum-time=7d
sudo find /var/log -name "*.gz" -delete

# 5. Проверить удалённые, но занятые файлы:
lsof +L1 | grep deleted

## 📋 Шпаргалка команд

| Задача | Команда |
| :--- | :--- |
| Общая нагрузка | `top`, `htop`, `uptime` |
| CPU по процессам | `ps aux --sort=-%cpu` |
| RAM по процессам | `ps aux --sort=-%mem` |
| Дисковая активность | `iostat -x 1`, `iotop` |
| Сетевые соединения | `ss -tulnp`, `netstat -tulpn` |
| Открытые файлы | `lsof -p <PID>` |
| Большие файлы | `find / -size +1G` |
| Логи службы | `journalctl -u service -n 100` |
| Очистка кэша | `echo 3 > /proc/sys/vm/drop_caches` |
| Inodes | `df -i` |


#⚠️ Частые ошибки

Не убивай процессы сразу — сначала пойми причину.

kill -9 — только если процесс не отвечает на kill -15.

Не очищай кэш в продакшене без понимания последствий.

Проверяй df -i — место есть, а inodes кончились.

Смотри dmesg — ошибки ядра могут быть причиной.
